# CTV Emulator — Аналіз ресурсів

> Дата: 2026-03-11 | Версія: 1.0.0

---

## 1. Пам'ять на 1 потік (worker process)

Архітектура: **Master** → `child_process.fork()` → **N Worker processes**, кожен із headless Chromium.

### Компоненти worker-а

| Компонент | RAM |
|---|---|
| Node.js процес (базовий) | 30–50 MB |
| Chromium headless (1 на worker) | 100–150 MB |
| BrowserContext + Page (1 сесія) | 30–50 MB |
| Shaka Player + відео буфери | 20–40 MB / сесію |

### Зведена таблиця

| Конфігурація | Пам'ять |
|---|---|
| **1 worker, 1 сесія** | **180–290 MB** |
| **1 worker, 10 сесій** (макс. per worker) | **450–700 MB** |
| **20 workers × 10 сесій** (макс. кластер) | **9–14 GB** |
| Master process | 50–100 MB |

### Ліміти

- Docker memory limit: **8 GB** (`docker-compose.yml`)
- Безпечний максимум при 8 GB: **8–12 workers × 5–8 сесій = 40–96 одночасних сесій**
- Auto-restart workers кожні 100 оброблених сесій для очистки пам'яті

## 2. Трафік на завантаження відео (1 сесія)

### Розбивка по типах запитів

| Тип | Розмір | % трафіку |
|---|---|---|
| **Рекламне відео** (MP4/HLS, 15–30с, 1080p) | 5–20 MB | ~75% |
| **Контентне відео** (DASH, 5 секунд) | 2–5 MB | ~20% |
| Shaka Player JS (локальний файл) | 0 KB | 0% |
| VAST XML (до 5 wrappers) | 5–30 KB | <0.3% |
| RTB bid request + response | 5–10 KB | <0.1% |
| Tracking pixels (~15 шт) | 30–50 KB | <0.4% |
| Win/billing/loss notices | 2–5 KB | <0.1% |

### Підсумок на 1 сесію

| | Мінімум | Типово | Максимум |
|---|---|---|---|
| Відео (ad + content) | 7 MB | 12 MB | 25 MB |
| Інше (RTB, VAST, tracking) | 50 KB | 80 KB | 120 KB |
| **Разом** | **~7 MB** | **~12 MB** | **~25 MB** |

> **~99% трафіку — це відео.**

## 3. Трафік через проксі: до і після оптимізації

### Було (до split-proxy)

Через проксі проходив **тільки RTB bid request**. VAST, tracking, win/billing — ішли напряму з IP сервера (що створювало невідповідність IP для анти-фрод систем).

### Стало (після split-proxy)

**Через проксі** (IP-критичні запити):
- RTB bid request/response
- VAST XML fetch (wrapper chain)
- Tracking pixels (impression, quartiles, complete, click)
- Win/billing/loss notices
- ClickThrough fetch

**Напряму** (без проксі):
- Рекламне відео (MP4/HLS) — завантажується Chromium
- Контентне відео (DASH) — завантажується Chromium

### Порівняння

| Сценарій | Проксі-трафік / сесію | Прямий трафік / сесію |
|---|---|---|
| **Все через проксі** (гіпотетичний) | 12–15 MB | 0 |
| **Split-proxy** (поточний) | **50–120 KB** | 12–15 MB |

**Економія: ~99.5% проксі-трафіку.**

### Масштаб

При повному навантаженні (200 сесій, цикл ~30с ≈ 24 000 сесій/год):

| | Все через проксі | Split-proxy | Економія |
|---|---|---|---|
| Проксі/год | 290–360 GB | **1.2–2.9 GB** | **~99%** |
| Проксі/день | 7–8.6 TB | **29–70 GB** | **~99%** |
| Проксі/місяць | 210–260 TB | **0.9–2.1 TB** | **~99%** |

## 4. Ризики split-proxy

| Ризик | Рівень | Пояснення |
|---|---|---|
| IP mismatch на відео CDN | Низький | CDN сервлять файли, не перевіряють IP проти bid request |
| Анти-фрод (IAS, DV, MOAT) на tracking | Немає | Tracking pixels йдуть через проксі — IP збігається |
| DSP перевірка nurl/burl IP | Немає | Win/billing notices йдуть через проксі |
| SSAI (server-side ad insertion) | Немає | Media URL з іншого домену, IP не звіряється |

## 5. Рекомендації

1. **Docker memory limit 8 GB замалий** для 200 сесій. Збільшити до 16 GB або зменшити `MAX_WORKERS`/`SESSIONS_PER_WORKER`
2. **Контентне відео грає 5с** — якщо мета тільки рекламні перегляди, можна скоротити або прибрати, зекономивши ~2–5 MB/сесію
3. **Shaka Player вже локальний** — економія ~523 KB/сесію (раніше завантажувався з CDN)
4. **Моніторинг**: dashboard показує real-time метрики, додати відстеження proxy traffic volume через network interceptor


TECHNICAL SPECIFICATION
Project: Headless CTV Emulator Platform
Version: 2.0 (Production Architecture)
Date: March 2026

1. GENERAL DESCRIPTION
The system is a scalable platform that emulates Connected TV (CTV) player behavior in headless mode. 
It reproduces OTT video playback, VAST advertising, OpenRTB bidding flows, and tracking events.

Use cases:
- Integration testing for SSP/DSP platforms
- Validation of VAST creatives
- Testing tracking events and ad delivery logic
- Load testing of video infrastructure
- Anti‑fraud testing

2. CORE CONCEPT
Each session simulates an isolated CTV device environment with:
- Playwright Chromium instance
- Shaka Player video playback
- RTB adapter
- VAST resolver
- Tracking engine
- Network telemetry collector

The platform must support at least 200 parallel sessions.

3. ARCHITECTURE
Master process:
- API server (Fastify)
- Session scheduler
- Metrics registry
- Worker manager

Worker process:
- Playwright browser
- Session instances
- Network instrumentation

Workers handle multiple sessions (default: 10 per worker).

4. TECH STACK
Node.js 20 LTS
TypeScript (strict mode)
Playwright (Chromium)
Shaka Player 4.x
Fastify API
Zod validation
Pino logging
Docker deployment
Prometheus metrics (prom-client)

5. SESSION STATE MACHINE
States:
CREATED → INITIALIZING → RTB_REQUESTING → VAST_RESOLVING → AD_LOADING → AD_PLAYING → CONTENT_PLAYING → STOPPING → STOPPED

Error states:
ERROR_VAST
ERROR_MEDIA
ERROR_NETWORK
ERROR_TIMEOUT

6. DEVICE EMULATION
Each session must emulate a CTV device including:
- OS (Android TV / Tizen / WebOS)
- Vendor and model
- Screen resolution
- Device ID / IFA
- IP and carrier
- Network type

7. OPENRTB REQUESTS
Bid requests follow OpenRTB 2.6 and include:
imp.video parameters
app.bundle
app.name
app.storeurl
device.ua
device.devicetype=7

8. VAST ENGINE
Supported formats:
- VAST 2.0
- VAST 3.0
- VAST 4.x

Supported structures:
- Inline
- Wrapper

Maximum wrapper depth: 5
Wrapper timeout: 3 seconds per hop.

9. AD TIMELINE ENGINE
Scheduler fires events based on ad duration.

Example for 20 second ad:
0s  impression
0s  start
5s  firstQuartile
10s midpoint
15s thirdQuartile
20s complete

Allowed timing jitter ±1.5 seconds.

10. TRACKING ENGINE
Events:
impression
start
firstQuartile
midpoint
thirdQuartile
complete
error

Each event must fire exactly once using idempotency keys.

11. NETWORK INSTRUMENTATION
Playwright intercepts:
request
response

rtb

media
tracking
content

All events logged to NDJSON logs.

12. PLAYER IMPLEMENTATION
File:
public/player.html

Video elements:
content-video
ad-video

~40MB per session
